Flow 01 · Setup

Setup & first run

Engram is two pieces that never overlap: the engram skill does all the thinking, and the engram-mcp server does all the touching of files. You point the server at a folder, register it once, and then talk to the skill.

Reading these snippets. Calls are written in Python form — write_note(path=…, content=…, overwrite=False) — to mirror the server's tool signatures one-to-one. In practice Claude issues them as MCP tool calls over stdio (FastMCP); you almost never type them yourself. Returns are shown as dicts and tagged # illustrative shape — confirmed on implementation: the spec pins tool names and arguments, not exact return schemas.

1 · Point Engram at a vault

The server reads and writes one folder, set by ENGRAM_VAULT. Omit it and Engram uses engram-data/. The vault is plain Markdown — Obsidian-compatible, hand-editable, no lock-in.

shell env
# Point Engram at a vault — omit to use the default: engram-data/
export ENGRAM_VAULT="$HOME/engram-data"

2 · Register the server (once)

The server speaks MCP over stdio, so Claude launches it as a child process. Register it in your Claude MCP config and forget it exists.

claude mcp config illustrative
{
  "mcpServers": {
    "engram": {
      "command": "engram-mcp",        # illustrative launch command
      "env": { "ENGRAM_VAULT": "~/engram-data" }
    }
  }
}

3 · Open the skill

Everything human happens through /engram. It also auto-triggers the moment you say "I learned…", so on most days you never type the command at all.

you /engram

On a brand-new vault there is nothing to list — folders are created lazily, only when a note needs them. A first peek is honest about that:

call over stdio
list_dir(path=".")
returns illustrative
{
    "path": ".",
    "folders": [],
    "notes": [],   # a fresh vault is empty; the tree below is a map, not a scaffold
}  # illustrative shape — confirmed on implementation
When to set ENGRAM_VAULT

Set it once per machine to keep your vault outside the repo (e.g. $HOME/engram-data). Leave it unset for a throwaway or per-project vault that lives at engram-data/ in the working directory.

Flow 02 · The hero flow

A learning session, end to end

This is the whole reason Engram exists. You type /engram and explain Tokio's work-stealing scheduler in your own words — no headings, no frontmatter, no thinking about Markdown. Engram decides where it belongs, captures your sentences, links it into what you already know, and asks you exactly one good question back.

you /engram "Each Tokio worker thread has its own local run queue. When a worker drains its queue it steals a batch of tasks from the back of another worker's queue instead of sitting idle. Tasks spawned from outside the runtime go on a global injector queue. The whole point is to spread load without a single global lock becoming the bottleneck."

Behind that one message, the skill runs a fixed sequence of MCP calls. It branches once — at dedupe — depending on whether a related note already exists:

1
Parse intent — topic, explanation, optional source / tags
Pure reasoning in the skill. No file is touched yet.
2
Dedupe — is this already in the vault?
search(query, limit) list_dir(folder)
No match → capture new
create_folder(path)
Lazy — only if the folder doesn't exist yet.
Match → deepen existing
append_note(…) / new title
The edge variant below.
3
Capture — fill the template strictly from your words
write_note(path, content, overwrite=False)
overwrite=False refuses to clobber — capture can never overwrite by accident.
4
Re-index — rewrite the folder's index.md
rebuild_index(folder)
Idempotent — a pure function of directory contents.
5
Connect — propose only real [[links]]
get_backlinks(note) search(query, limit)
Links are drawn from actual hits, then you confirm them — never invented.
6
Mentor — exactly one question, never a lecture

The calls, with their returns

First the dedupe. Nothing matches, and the Rust folder doesn't exist yet — both facts point at "create new":

call dedupe
search(query="work stealing scheduler", limit=5)
returns illustrative
{
    "query": "work stealing scheduler",
    "hits": [],   # nothing yet — safe to create a fresh note
}  # illustrative shape — confirmed on implementation
call dedupe
list_dir(path="Engineering/Languages/Rust")
returns illustrative
{
    "path": "Engineering/Languages/Rust",
    "error": "not_found",   # lazy — the folder is born with the first note
}  # illustrative shape — confirmed on implementation

So the skill creates the folder (nested, lazily) and writes the note. create_folder reports every level it had to make:

call lazy create
create_folder(path="Engineering/Languages/Rust")
returns illustrative
{
    "created": [
        "Engineering",
        "Engineering/Languages",
        "Engineering/Languages/Rust",
    ],
}  # illustrative shape — confirmed on implementation
call capture
write_note(
    path="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    content=note_md,        # the filled template, shown below
    overwrite=False,        # refuse to clobber an existing note
)
returns illustrative
{
    "path": "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    "created": True,
    "title": "Tokio Work-Stealing Scheduler",
}  # illustrative shape — confirmed on implementation

Then the folder's index.md is regenerated, and the skill looks for genuine neighbors to link:

call re-index + connect
rebuild_index(folder="Engineering/Languages/Rust")
get_backlinks(note="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md")
search(query="async runtime", limit=5)
returns illustrative
{
    "index": "Engineering/Languages/Rust/index.md",
    "children": {"folders": 0, "notes": 1},
    "updated": True,
}  # illustrative shape — confirmed on implementation

# search surfaces a real note → the skill proposes [[Async Runtime]]
# and offers [[Green Threads vs OS Threads]] as a candidate you confirm.

Only links that resolve to real hits are proposed. A topic with no note yet stays a candidate until you say yes — Engram never fabricates a connection to make the graph look fuller.

The note that lands

Here is note_md — the Standard Note Template, filled from the explanation above. The Explanation section is your sentences, lightly formatted; nothing is rewritten into textbook prose, and the empty Examples section is left honestly empty rather than invented.

tokio-work-stealing-scheduler.md markdown + frontmatter
---
title: Tokio Work-Stealing Scheduler
category: Languages/Rust
status: learning
confidence: 2
created: 2026-06-28
updated: 2026-06-28
tags: [rust, tokio, concurrency, async, scheduler]
source:
---

# Summary
Tokio keeps every worker thread busy with per-worker local run queues; an
idle worker steals a batch of tasks from another worker instead of waiting.

# Explanation
Each worker thread has its own local run queue. When a worker drains its
queue, it steals a batch of tasks from the back of another worker's queue
instead of sitting idle. Tasks spawned from outside the runtime go on a
global injector queue. The whole point is to spread load without a single
global lock becoming the bottleneck.

# Examples
(none provided yet)

# Related Topics
[[Async Runtime]]
[[Green Threads vs OS Threads]]

# Questions I Still Have
- How large is a "batch" when one worker steals from another?

# Revision Questions
- Explain work-stealing without looking.
- What problem does the per-worker local queue solve?
- When would the global injector queue itself become the bottleneck?

A note can also carry a per-note metadata block, maintained as you revisit it over time:

first_learned last_reviewed last_used projects related difficulty status

And then — one question

The session closes with Mentor Mode: exactly one follow-up, or one pointed misunderstanding, or one next-topic nudge. Never a lecture.

One to sit with — when every worker's local queue is already full, where does a task spawned from outside the runtime go, and what keeps that injector queue from becoming the single global lock you just designed around? — Engram, Mentor Mode (one question, then it stops)

Edge variant — the note already exists

Run the same session a second time and dedupe finds the note. write_note with the default overwrite=False refuses, loudly:

call capture (retry)
write_note(
    path="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    content=note_md,
    overwrite=False,
)
returns illustrative
{
    "error": "note_exists",
    "path": "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
}  # illustrative shape — confirmed on implementation

The skill now chooses deliberately. If this is the same concept going deeper, it appends a section:

call deepen in place
append_note(
    path="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    content="\n## A worker steals roughly half of another's queue, not one task.\n",
)

If it's a genuinely distinct topic, it writes under a new title instead:

call distinct topic
write_note(
    path="Engineering/Languages/Rust/tokio-task-budgeting.md",
    content=note_md,
    overwrite=False,
)
Append vs new title vs overwrite

Append when it's the same idea deepened. New title when it's a separate concept that merely sounds related. overwrite=True is reserved for a deliberate replacement you asked for — it is never the default escape hatch.

Flow 03 · Retrieval

Rediscovery, months later

The payoff is future-you finding present-you's thinking fast. Three tools cover it: search when you remember words, find_by_metadata when you remember a property, and get_backlinks when you want the web around a note.

Search — ranked, and it tells you why

Hits are ranked by where the match landed: title > alias > tag > heading > content > linked-notes. Each hit names its matched field, so a result is never a mystery.

call
search(query="work stealing", limit=5)
returns illustrative
{
    "query": "work stealing",
    "hits": [
        {
            "path": "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
            "field": "title",     # strongest signal → ranks first
            "snippet": "Tokio Work-Stealing Scheduler",
            "score": 0.98,
        },
        {
            "path": "Engineering/Foundations/Concurrency/run-queues.md",
            "field": "content",   # weaker signal → ranks lower
            "snippet": "…steals a batch from the back of another worker's queue…",
            "score": 0.41,
        },
    ],
}  # illustrative shape — confirmed on implementation

Find by metadata — when you remember a property

You don't always remember words. Sometimes you remember "that Rust thing," or "the stuff I only half-understood." find_by_metadata filters on frontmatter:

call by tag
find_by_metadata(field="tag", value="rust")
returns illustrative
{
    "field": "tag",
    "value": "rust",
    "notes": [
        "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
        "Engineering/Languages/Rust/pin-and-unpin.md",
    ],
}  # illustrative shape — confirmed on implementation

The filtered variant turns your own honesty into a review queue. The confidence scale runs 0–5; querying for 2 surfaces everything you marked "understand the basics" and no further:

0never seen
1recognize term
2understand basics
3can explain
4have built with it
5can teach it
call by confidence — the review queue
find_by_metadata(field="confidence", value=2)
returns illustrative
{
    "field": "confidence",
    "value": 2,
    "notes": [
        "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    ],
}  # illustrative shape — confirmed on implementation

Backlinks — the web around a note

To see what points at a note rather than the note itself, ask for its backlinks:

call
get_backlinks(note="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md")
returns illustrative
{
    "note": "Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    "backlinks": [
        "Engineering/Languages/Rust/async-runtime.md",
        "Engineering/Projects/voice-agent-runtime.md",
    ],
}  # illustrative shape — confirmed on implementation
Which retrieval tool

search when you remember roughly what you wrote. find_by_metadata when you remember a property — a tag, a status, a confidence level. get_backlinks when the note isn't the point and the connections are.

Flow 04 · Upkeep

Maintaining the vault

Knowledge moves. A note you filed under Languages/Rust/ turns out to be a concurrency idea; a title reads better reworded. move and rename_note relocate things; rebuild_index keeps the navigation honest afterward.

Move, then re-index both ends

An index is a pure function of its folder's contents, so after a move you rebuild both the old and new parents:

call relocate
move(
    src="Engineering/Languages/Rust/tokio-work-stealing-scheduler.md",
    dest="Engineering/Foundations/Concurrency/tokio-work-stealing-scheduler.md",
)
rebuild_index(folder="Engineering/Languages/Rust")            # note left here
rebuild_index(folder="Engineering/Foundations/Concurrency")   # note landed here

Rename a title, then re-index

call rename + re-index
rename_note(
    path="Engineering/Foundations/Concurrency/tokio-work-stealing-scheduler.md",
    new_title="Work-Stealing Scheduler (Tokio)",
)
rebuild_index(folder="Engineering/Foundations/Concurrency")

Peek without reading the whole note

Retrieval discipline: skim structure cheaply instead of pulling full bodies into context. list_dir, list_headings and get_metadata are the three lenses.

call outline only
list_headings(note="Engineering/Foundations/Concurrency/tokio-work-stealing-scheduler.md")
returns illustrative
{
    "note": "…/tokio-work-stealing-scheduler.md",
    "headings": [
        "Summary", "Explanation", "Examples",
        "Related Topics", "Questions I Still Have", "Revision Questions",
    ],
}  # illustrative shape — confirmed on implementation
call frontmatter only
get_metadata(note="Engineering/Foundations/Concurrency/tokio-work-stealing-scheduler.md")
returns illustrative
{
    "title": "Work-Stealing Scheduler (Tokio)",
    "category": "Foundations/Concurrency",
    "status": "learning",
    "confidence": 2,
    "created": "2026-06-28",
    "updated": "2026-06-28",
    "tags": ["rust", "tokio", "concurrency", "async", "scheduler"],
    "source": None,
}  # illustrative shape — confirmed on implementation

For a heavier reorg of a whole branch, rename_folder(path, new_name) and move(src, dest) work on folders too — same rule applies: rebuild the touched indexes when you're done.

When to call rebuild_index

After every add, move, or rename — and whenever an index looks stale. It's idempotent: calling it twice costs nothing and the output never drifts, so over-calling is always safe.

Flow 05 · Edges

Safety behaviors you'll meet

Engram fails loudly and recoverably, and never destroys your data. Four edges you'll hit, and exactly what you see at the boundary each time.

note_exists

The overwrite guard

repl illustrative
write_note(path="…/tokio-work-stealing-scheduler.md", content=note_md, overwrite=False)
# → {"error": "note_exists", "path": "…/tokio-work-stealing-scheduler.md"}
You can't lose a note to a stray write. The skill must pick a path on purpose — append, a new title, or an explicit overwrite=True.
.trash/

Soft delete — recoverable

repl illustrative
delete_note(path="Engineering/Languages/Rust/pin-and-unpin.md")
# → {"deleted": True, "moved_to": ".trash/Engineering/Languages/Rust/pin-and-unpin.md"}

# recover by moving it back out of .trash/
move(src=".trash/Engineering/Languages/Rust/pin-and-unpin.md",
     dest="Engineering/Languages/Rust/pin-and-unpin.md")
Nothing is hard-deleted in v0.1. delete_note moves the file to .trash/; it stays recoverable with a plain move.
frontmatter_error

A hand-edit typo never crashes

repl illustrative
get_metadata(note="Engineering/Languages/Rust/lifetimes.md")
# → {
#       "frontmatter_error": True,                       # the YAML didn't parse…
#       "raw_frontmatter": "tags: [rust, lifetimes\nconfidence: 3",  # …handed back as-is
#   }
You fat-finger the YAML, the parser stays lenient: it returns the raw block with a frontmatter_error flag instead of throwing. The note still indexes by body, headings and links — only metadata queries skip it until you fix the typo by hand.
path escape

The vault is a sealed root

repl illustrative
read_note(path="../../etc/passwd")
# → {"error": "path_escape"}   # illustrative code — name confirmed on implementation
Every path resolves under ENGRAM_VAULT first. A .. or absolute path is rejected before any filesystem touch; the skill surfaces the error and doesn't retry blindly.
Why it reads this way. Structured, named errors let the skill react deliberately — choose a new title, recover from .trash/, fix a typo — rather than paper over a failure. The boundary is the product: the server fails honestly, and the reasoning lives entirely in the skill.

API cheat sheet
